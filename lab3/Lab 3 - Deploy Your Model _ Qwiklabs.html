Lab overview

In the previous labs, you explored the basics of setting up, testing, and evaluating your trained machine learning model. Now, you will explore the basics of deploying the trained model into production.

In this lab, you will update AWS CodePipeline to have a manual approval step and a step that launches an AWS CloudFormation template. This template will take the trained model and create a production endpoint for you to use. Depending on your business use case, you will need to extend this setup to meet your needs.

Some enhancements to consider include:

  • Deploying a new version of the model
  • Deploying the model in a different AWS Account
  • Updating the artifact store to include the production endpoint information
  • Adding an API Gateway to access the endpoint
  • Moving the endpoint to a VPC to restrict external access

Objectives

After completing this lab, you will be able to:

  • Extend CodePipeline by adding a manual approval step
  • Invoke AWS CloudFormation with CodePipeline
  • Pass parameters from CodePipeline to AWS CloudFormation

Prerequisites

This lab requires:

  • Access to a notebook computer with Wi-Fi and Microsoft Windows, Mac OS X, or Linux (Ubuntu, SuSE, or Red Hat)
  • The lab environment is not accessible using an iPad or tablet device, but you can use these devices to access the student guide.
  • For Microsoft Windows users: Administrator access to the computer
  • An Internet browser such as Chrome, Firefox, or IE9 (previous versions of Internet Explorer are not supported)

Duration

This lab will require 60 minutes to complete.

Now that you understand how to manage a machine learning pipeline, you will learn one method to deploy a trained model into production. Your business goals will determine the deployment strategy that meets your needs.

Task 1: Manually creating an Amazon SageMaker endpoint

Manual deployment of a trained model is possible; however, it does not scale. To understand the deployment process, you will first manually deploy a trained model. You will then automate the deployment process in a later task. This will help you to understand how to take a stored, trained model and deploy it (once or many times) to meet your business goals. Storing trained models for deployment helps to maintain data continuity, rather than attempting to retrain a model. Using stored models may also reduce the cost of rerunning long training jobs.

The following workflow was established in previous labs. You will use the output from Amazon SageMaker, stored in Amazon Simple Storage Service (Amazon S3), to manually create a SageMaker endpoint.

In this task, you will access the S3 console to identify where the model artifact is, and then you will use this artifact to manually create an endpoint.

  1. In the AWS Management Console, on the Services menu, choose S3.

  2. Choose the name of the bucket that contains modelArtifactBucket.

This bucket contains all the artifacts from AWS CodeBuild and Amazon SageMaker. The model registry, stored in Amazon DynamoDB, provides the information needed to identify where each training job artifact is located. As you build your infrastructure you want to consider the implications of commingling artifacts into a single bucket. Depending on your use case, you will need to choose a data storage structure that is easy for you to maintain. For our simple example here, we will only create 2-4 artifacts that will be easy to identify in a commingled bucket. It is always important to think about what decisions will aid long-term maintenance.

You should see multiple objects in the artifact bucket. Each AWS CodePipeline pipeline run will create an object in this bucket that contains the artifacts from that run.

In addition, one object has a name that starts with Job-####. This is the output from the training job as specified by the Step Functions state machine.

  1. Choose the name of the object that starts with Job-####.

  2. Choose the object named output.

  3. Choose the object named model.tar.gz.

The model object properties are displayed.

For learning purposes, you can download the model.tar.gz file and use it to deploy an endpoint in a different account, or use it directly with your application code. To maintain security, you should consider storing model artifacts in a restricted bucket that does not allow modification. Or, for deployment into another account, you may establish cross-account privileges to allow model deployment. In this task, you will reference this trained model and manually deploy it in this account.

Task 2: Update the AWS CodePipeline to include a manual approval step

Automated processes are consistent and efficient, but there are times when human judgment is critical for business or quality reasons. AWS CodePipeline can help you add steps that require manual approval.

The final architecture will require you to add to the model deployment pipeline, and will require you to add a CloudFormation template that creates a new model and endpoint.

You have developed most of this architecture in previous labs, so you will focus on updating the training CodePipeline here. In this task, you will create a new deployment stage and add a manual approval step that will send a notification requesting approval.

  1. In the AWS Management Console, on the Services menu, choose CodePipeline.

  5. For Stage name, enter

  9. For Action provider, choose Manual approval.

Task 3: Update the AWS CodePipeline to include an Amazon CloudFormation step

Now that the manual approval step has been created, you will add a CloudFormation step that will automate the deployment after approval.

  1. In the Deploy card, choose the bottom Add action group.

  2. For Action name, enter

  3. For Action provider, choose AWS CloudFormation.

  4. For Input artifacts, choose ModelSourceOutput.

Note ModelSourceOutput was defined in the CodePipeline source stage. In this example, we will be using the modeCode repository that the model code is stored in to store the deployment script. Depending on your needs, you could use a separate CodeCommit repository, upload the deployment script to an Amazon S3 bucket, or use a third-party source.

  1. For Action mode, choose Create or update a stack.

  2. For Stack name, enter:

This will automatically generate a unique stack name each time the pipeline is run. #{codepipeline.PipelineExecutionId} will add the execution id that CodePipeline generates when invoked. In your own production environment, you will need to decide if deploying a new stack is appropriate for your use case. You may want to update a stack instead of creating a new one.

  1. In the Template section for Artifact name, choose ModelSourceOutput.

  2. In the Template section for File name, enter

In the next task, you will create a new file named deploy.yaml and commit it to the model repo. This is how you define the source for the production deployment. In your own production environment, you will have to consider your entire architecture to determine the best way to integrate this step.

  1. For Role name, choose the role that contains ProductionCloudFormationRole.

This role has been pre-created for you to maintain the best-practice security principal of least privileges.

  1. Choose Advanced to expand the section.

Adding parameter overrides sends unique configuration information from CodePipeline to be passed to CloudFormation. In this way, you can write a dynamic CloudFormation template that does not need to be updated each time you train a new model.

  1. Copy the following JSON block to a notepad:
{
  "S3Bucket": "<modelArtifactBucket>", 
  "ProductionModelRole": "<ProductionModelRole>", 
  "TrainingJob": "Job-#{codepipeline.PipelineExecutionId}", 
  "ECRImage": "<ECRModelRepo>:#{codepipeline.PipelineExecutionId}"
}

  1. Update modelArtifactBucket, ProductionModelRole, and ECRModelRepo with the values from the left side of the labs page.

  2. Copy the resulting JSON block and paste it into the Parameter overrides box.

  3. Choose Done.

  4. At the top of the page, choose Save.

  5. On the Save pipeline changes message window that appears, choose Save.

You have now configured CodePipeline to have a manual approval step, and a CloudFormation step that will automate the deployment.

Task 5: Update the model repo to include a deployment script

In this task, you will add a deployment template to the CodeCommit repository. This will automatically invoke a build so you can test the production deployment process.

  1. In the AWS Management Console, on the Services menu, choose CodeCommit.

  2. Choose the name of the repository that includes modelCode.

  3. Choose Add file and select Create file.

  4. Paste the following code into the text box:

AWSTemplateFormatVersion: 2010-09-09
Description: 'Deploy trained ML model to production'

# Parameters are values set when the template is launched.
# The values for these parameters will be passed in by 
# CodePipeline as defined in the parameter overrides
Parameters:
  S3Bucket:
    Description: 'Name of the S3 bucket where the model artifact is stored'
    Type: String
    Default: 'none'
  ProductionModelRole:
    Description: 'The Arn of the ProductionModelRole used by CloudFormation'
    Type: String
    Default: 'none'
  TrainingJob:
    Description: 'The name of the training job'
    Type: String
    Default: 'none'
  ECRImage:
    Description: 'The ERC image URI:Tag'
    Type: String
    Default: 'none'
  InstanceType:
    Description: 'The instance type for the endpoint'
    Type: String
    Default: 'ml.m5.large'
    AllowedValues:
      - 'ml.m5.large'
      - 'm1.t2.large'
      - 'm1.t2.medium'

# The resources section defines what services will be launched
# by the template.
Resources:
  # Create an Amazon SageMaker model
  ProductionModel:
    Type: AWS::SageMaker::Model
    Properties: 
      ExecutionRoleArn: !Ref ProductionModelRole
      PrimaryContainer: 
        Image: !Ref ECRImage
        Mode: SingleModel
        ModelDataUrl: !Sub s3://${S3Bucket}/${TrainingJob}/output/model.tar.gz

  # Create an endpoint configuration
  ProductionEndpointConfiguration:
    Type: AWS::SageMaker::EndpointConfig
    Properties: 
      ProductionVariants: 
        - InitialInstanceCount: 1
          InitialVariantWeight: 1.0
          InstanceType: !Ref InstanceType
          ModelName: !GetAtt ProductionModel.ModelName
          VariantName: !GetAtt ProductionModel.ModelName

  # Create an endpoint
  ProductionEndpoint:
    Type: "AWS::SageMaker::Endpoint"
    Properties:
      EndpointConfigName: !GetAtt ProductionEndpointConfiguration.EndpointConfigName

# Outputs are values that are displayed in the outputs section of
# AWS CloudFormation.
Outputs:
  EndpointId:
    Value: !Ref ProductionEndpoint
  EndpointName:
    Value: !GetAtt ProductionEndpoint.EndpointName
  EndpointURL:
    Value: !Sub https://runtime.sagemaker.${AWS::Region}.amazonaws.com/endpoints/${ProductionEndpoint}/invocations

This template will take the parameters passed in by CodePipeline (which you setup in the previous task) and use them to create a new model, endpoint configuration, and endpoint the same way you manually created it earlier.

A properly written template will work in any account because the parameters are passed in by CodePipeline. This means that you do not have to make any replacements in the code, which is one advantage of this method.

  1. In the Commit changes to main section, fill in the following:
  • File name:
  • Author name: enter your name
  • Author email: enter your email address
  • Commit message:
  1. Choose Commit changes.

The CodePipeline will detect a change to the repo and automatically start running.

Task 6: Review, approve, and test the model

Now that the model has been trained and deployed to a staging endpoint, a manual approval step must be completed. This is the final check before the model is deployed to the production endpoint. Manual approval is not needed for every use case, and you will need to decide if manual approval is appropriate for your own production environment. You should have received a notification from the SNS topic you set up earlier, notifying you that the production model is ready for approval.

  1. In the AWS Management Console, on the Services menu, choose CodePipeline.

  2. Choose the name of the pipeline that contains deployModelPipeline.

  3. Wait for the pipeline to reach the Deploy stage.

You should receive an email that asks you to approve the deployment. You may either follow the link in the email, or choose Approve from the Deploy stage in the CodePipeline console.

  1. In the Deploy stage, choose Review.

  2. In the comment box, enter

  3. Choose Approve.

The AWS CloudFormation template will now be launched for the production endpoint.

  1. In the AWS Management Console, on the Services menu, choose CloudFormation.

  2. Choose the name of the stack that starts with Production-####.

  3. Choose the Parameters tab.

Notice that the parameters have been automatically filled in for you, with no complex copy and paste needed. This will also work every time the model code is updated, removing the need for human interaction beyond the manual approval.

The CloudFormation template will take 10-15 minutes to deploy. While the stack is deploying, you may choose to review the Amazon SageMaker model, endpoint configuration, and endpoint pages to see how resources are deployed over time.

  1. When the stack is listed as CREATE_COMPLETE, choose the Outputs tab.

The outputs have been configured to list the endpoint connection information. The outputs can be used to automatically update code or configuration parameters as needed.

  1. Copy the EndpointName value to your notepad to use in the next task.

The EndpointName name is the DNS endpoint that you use to connect to the endpoint, send requests to, and receive predictions.

You have fully automated the deployment of your model. You may now extend this basic example to meet your exact needs.
